Why LLM Inference Needs Specialized Observability

If you have operated a Kubernetes cluster for any length of time, you already know the standard monitoring playbook: CPU utilization, memory consumption, pod restart counts, network throughput, and HTTP error rates. These metrics serve traditional microservices well. A request arrives, consumes some CPU and memory, touches a database, and returns a result within a few milliseconds. When something goes wrong, CPU spikes or memory leaks or error rates climb, and your dashboards tell you exactly where to look. This playbook is well-established, battle-tested, and catastrophically insufficient for LLM inference workloads.

The fundamental problem is that standard Kubernetes monitoring was designed for stateless request-response services. LLM inference operates on an entirely different computational model. A single inference request can consume gigabytes of GPU memory, run for seconds or minutes, and involve complex internal state management that has no analog in traditional web services. The failure modes are correspondingly alien to anyone accustomed to cloud-native operations, and the metrics that matter are ones that Kubernetes does not even know exist.

Consider the most deceptive metric in GPU computing: utilization. Your DCGM exporter might report 92% GPU utilization across all pods, and your instinct tells you everything is running smoothly. But behind that number, every single request might be performing cold prefill computations because your KV cache hit rate has collapsed. Users are experiencing five-second time-to-first-token latencies while your dashboards show green across the board. The GPU is busy. It is just busy doing redundant work that a properly functioning cache would have eliminated entirely.

LLM inference introduces failure modes that have no analog in traditional service architectures. KV cache eviction storms occur when memory pressure forces the eviction of cached key-value pairs, which triggers a devastating cascade: subsequent requests that would have been cache hits now become cold prefills, consuming more GPU compute, which increases memory pressure further, which triggers more evictions. This positive feedback loop can take a healthy cluster from sub-100ms TTFT to multi-second TTFT in under a minute, and standard monitoring will not register anything unusual until users start filing complaints.

Adapter swap thrashing is another uniquely LLM failure mode. When you serve multiple LoRA adapters on the same base model, the system needs to load and unload adapter weights as requests arrive for different fine-tuned variants. If your routing layer is not adapter-aware, you can end up in a situation where every request triggers an adapter swap, turning what should be a few milliseconds of overhead into a constant drag on throughput. Prefill/decode imbalance is yet another invisible killer: if too many requests are in the compute-intensive prefill phase simultaneously, the decode throughput for in-flight requests collapses because prefill operations consume disproportionate GPU compute and memory bandwidth. Head-of-line blocking in continuous batching systems adds still another layer of complexity. vLLM uses continuous batching to process multiple requests simultaneously, but a single request with a massive context window can monopolize the attention computation and starve shorter requests of GPU cycles. Without metrics that distinguish between prefill and decode phases, between cache hits and cache misses, between adapter loads and steady-state inference, you are operating blind in a system that fails in ways your existing monitoring was never designed to detect.

Key Insight: GPU utilization is not a proxy for inference quality. A cluster can show 95% GPU utilization while delivering terrible user experience because all that compute is spent on redundant prefill operations that proper cache-aware routing would have eliminated.

The Key Metrics That Matter

Effective LLM inference monitoring requires a fundamentally different set of metrics than traditional service monitoring. The metrics that matter are those that capture the unique computational characteristics of autoregressive token generation, KV cache behavior, and GPU memory management. There are six core metrics that tell you almost everything you need to know about system health and user experience.

Time-to-First-Token (TTFT)

TTFT measures the elapsed time from when a request arrives at the inference server to when the first token of the response is generated. This metric directly reflects the prefill latency, which is the dominant cost in LLM inference for long-context requests. TTFT is the metric that users feel most acutely: it is the pause between pressing send and seeing the model begin to respond. In a streaming interface, it is the silence before the first word appears. You need to track TTFT at multiple percentiles: p50 gives you the typical user experience, p95 reveals the tail latency that affects one in twenty users, and p99 exposes the worst-case behavior that drives user complaints and SLA violations. A healthy cluster serving Llama 3.1 70B should target p50 TTFT under 500ms and p99 TTFT under 3 seconds for typical prompt lengths. When p99 TTFT exceeds 5 seconds, something is wrong. When it exceeds 10 seconds, your users are actively churning. A cache hit can reduce TTFT from seconds to milliseconds, which is why cache-aware routing is the single most impactful optimization in the entire stack.

Inter-Token Latency (ITL)

ITL measures the time between consecutive tokens during the decode phase. While TTFT captures the initial wait, ITL determines the perceived streaming speed. Users are surprisingly sensitive to ITL variation: consistent 30ms ITL feels smooth and natural, while variable ITL that oscillates between 10ms and 200ms feels stuttery and broken even if the average is identical. ITL spikes typically indicate contention in the decode phase, often caused by new prefill operations being scheduled into the continuous batch, by GPU memory bandwidth saturation, or by thermal throttling. Track ITL at p50 and p99, and alert on p99 ITL exceeding 100ms. The distinction between TTFT and ITL matters because they have different root causes and different remediation strategies: TTFT problems are usually cache or queue related, while ITL problems are usually contention or hardware related.

Throughput: Tokens per Second

Throughput needs to be decomposed into prefill throughput and decode throughput, because they compete for different GPU resources and have different scaling characteristics. Prefill throughput is compute-bound and benefits from tensor parallelism and larger batch sizes. Decode throughput is memory-bandwidth-bound and benefits from higher batch sizes but is constrained by the autoregressive nature of generation. Aggregate tokens-per-second is useful for capacity planning and cost calculations, but the prefill/decode decomposition is essential for diagnosing performance issues. A sudden drop in decode throughput while prefill throughput remains constant is a strong signal that too many requests are entering the prefill phase simultaneously and starving the decode pipeline.

KV Cache Hit Ratio

This is arguably the single most important metric for an llm-d cluster. The KV cache hit ratio measures the fraction of incoming request tokens that are served from cached key-value pairs rather than computed from scratch during prefill. In a well-tuned cluster with cache-aware routing, you should see hit ratios above 60% for workloads with prompt reuse, such as chatbots using system prompts, RAG applications with repeated document contexts, or code completion with shared file contexts. A hit ratio below 40% means your routing layer is not effectively directing requests to pods that have cached their prefix, and you are paying a massive latency and compute tax for redundant prefill operations. Investigate whether the EPP cache affinity weight is too low, whether your cache size is too small and causing premature evictions, or whether your traffic genuinely lacks prefix overlap.

Queue Depth and GPU Memory Pressure

Queue depth measures how many requests are waiting to be processed at each inference pod. In continuous batching systems, a small non-zero queue is normal and healthy, but persistent queue depths above 20-30 indicate that your cluster is under-provisioned or that a few pods are hoarding traffic due to routing imbalances. Watch for queue depth asymmetry across pods: if one pod has a queue of 60 while others sit at 2, your load balancing is misconfigured. GPU memory utilization needs to be tracked separately from GPU compute utilization, because they have different failure thresholds and different remediation paths. Memory pressure above 90% triggers KV cache evictions, which degrades cache hit ratios, which increases compute demand, which increases memory pressure further. This is the eviction storm feedback loop, and catching it early requires alerting on GPU memory utilization well before it reaches the critical threshold where evictions begin cascading.

Warning: Do not confuse GPU compute utilization with GPU memory utilization. You can have 50% compute utilization and 95% memory utilization, which means you are one large-context request away from triggering KV cache evictions and a latency cascade.

Building a Monitoring Stack

The foundation of LLM inference observability is Prometheus for metrics collection and Grafana for visualization. vLLM exposes a comprehensive metrics endpoint at /metrics that provides Prometheus-formatted metrics covering request latencies, token throughput, cache statistics, and internal queue state. The monitoring stack architecture is straightforward: each vLLM pod exposes metrics on a configurable port, Prometheus scrapes them on a regular interval, and Grafana dashboards query Prometheus to render real-time and historical views of cluster behavior. You do not need to instrument anything yourself because vLLM provides all the metrics out of the box.

The first step is configuring Prometheus to discover and scrape your vLLM pods. If you are running on Kubernetes with the Prometheus Operator, the cleanest approach is to use a ServiceMonitor resource that automatically discovers pods based on label selectors. This eliminates the need to manually maintain scrape target lists as pods scale up and down, which is critical in environments where horizontal pod autoscaling is active.

# ServiceMonitor for vLLM inference pods
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: vllm-inference-monitor
  namespace: llm-d
  labels:
    release: prometheus
spec:
  selector:
    matchLabels:
      app: vllm-inference
  endpoints:
    - port: metrics
      interval: 15s
      path: /metrics
      scrapeTimeout: 10s
  namespaceSelector:
    matchNames:
      - llm-d

If you are not using the Prometheus Operator, you can configure static scrape targets or use Kubernetes service discovery directly in your Prometheus configuration file. The key parameters are the scrape interval, which should be 15 seconds for production LLM workloads to capture rapid changes in cache state and queue depth, and the scrape timeout, which should be set to 10 seconds to avoid marking healthy but busy pods as unresponsive during heavy inference loads.

# prometheus.yml scrape config for vLLM pods
scrape_configs:
  - job_name: 'vllm-inference'
    kubernetes_sd_configs:
      - role: pod
        namespaces:
          names:
            - llm-d
    relabel_configs:
      - source_labels: [__meta_kubernetes_pod_label_app]
        regex: vllm-inference
        action: keep
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_port]
        action: replace
        target_label: __address__
        regex: (.+)
        replacement: ${1}:8000
      - source_labels: [__meta_kubernetes_pod_name]
        target_label: pod
      - source_labels: [__meta_kubernetes_pod_label_model]
        target_label: model

Dashboard organization matters more than people realize. A common mistake is building a single monolithic dashboard with fifty panels that no one can parse during an incident. Instead, organize your dashboards into a clear hierarchy: a top-level overview dashboard that shows cluster-wide health at a glance, a pod-level dashboard that shows detailed metrics for individual inference servers, a cache dashboard focused on KV cache behavior and routing effectiveness, and a cost dashboard that tracks GPU utilization efficiency and cost-per-token. Each dashboard should answer a specific question. The overview dashboard answers "is the cluster healthy right now?" The pod dashboard answers "why is this specific pod underperforming?" The cache dashboard answers "is our routing layer doing its job?" This separation keeps each dashboard focused and scannable rather than overwhelming operators with an undifferentiated wall of charts during a high-stress incident.

PromQL Queries for Critical Dashboards

The power of Prometheus lies in PromQL, its query language. For LLM inference monitoring, you need queries that go beyond simple rate calculations to capture the nuanced behavior of autoregressive generation, cache dynamics, and multi-model serving. Here are the essential queries for every llm-d production dashboard, with explanations of what each measures and how to use it effectively in Grafana panels.

TTFT Percentiles

The most critical query in your entire monitoring stack tracks time-to-first-token at the 99th percentile. This single number tells you more about user experience than any other metric in your system.

# P99 Time-to-First-Token
histogram_quantile(0.99,
  rate(vllm_request_ttft_seconds_bucket[5m])
)

# P95 and P50 for comparison
histogram_quantile(0.95, rate(vllm_request_ttft_seconds_bucket[5m]))
histogram_quantile(0.50, rate(vllm_request_ttft_seconds_bucket[5m]))

Display p50, p95, and p99 on the same Grafana panel as overlaid time series. When p99 diverges significantly from p50, it indicates that most requests are fast but a subset is hitting a pathological code path, typically cold prefill on cache misses or extended queue wait times due to contention.

Token Throughput

# Aggregate token generation rate across all pods
sum(rate(vllm_tokens_generated_total[1m]))

# Per-pod token throughput for identifying stragglers
rate(vllm_tokens_generated_total[1m])

Display the per-pod query as a stacked area chart to visualize how throughput is distributed. If one pod is consistently generating fewer tokens than its peers, it may have a hardware issue, a cache problem, or a routing imbalance that is directing disproportionate traffic elsewhere.

KV Cache Hit Rate

# Cache hit ratio (percentage of prefix tokens served from cache)
sum(rate(vllm_cache_hits_total[5m])) /
(sum(rate(vllm_cache_hits_total[5m])) + sum(rate(vllm_cache_misses_total[5m])))
* 100

Format this as a percentage in Grafana and set color thresholds: green above 60%, yellow between 40% and 60%, red below 40%. This single panel tells you whether the EPP's cache-aware routing is delivering value or whether requests are being scattered across pods without regard for cache affinity.

GPU Memory, Queue Depth, Request Rate, and Error Rate

# GPU memory utilization per pod (via DCGM exporter)
DCGM_FI_DEV_FB_USED / DCGM_FI_DEV_FB_TOTAL * 100

# Queue depth across all pods
sum(vllm_num_requests_waiting)

# Request rate by model (for multi-model deployments)
sum by (model) (rate(vllm_request_total[5m]))

# Error rate (5xx and OOM)
sum(rate(vllm_request_errors_total[5m])) /
sum(rate(vllm_request_total[5m])) * 100

These queries form the backbone of your operational dashboards. Each one answers a specific question: Are users waiting too long? Are we generating tokens fast enough? Is the cache working? Are GPUs running out of memory? Are requests piling up? Which models are getting the most traffic? Are we returning errors? Layer these queries into Grafana panels with appropriate color-coded thresholds and you have a monitoring stack that exposes the true health of your LLM inference cluster rather than the misleading picture painted by generic Kubernetes metrics.

Alerting Strategy

Not everything that belongs on a dashboard deserves an alert. Alert fatigue is the fastest way to ensure that real incidents get ignored because operators have been conditioned to dismiss notifications as noise. The distinction is simple: dashboards are for exploration and context during investigation, alerts are for situations that require human intervention within a defined time window. If a metric being out of range does not require someone to take action, it belongs on a dashboard panel, not in an alerting rule.

For LLM inference workloads, there is a clear hierarchy of alertable conditions. The thresholds below are calibrated for production llm-d deployments serving 70B-parameter models on A100 or H100 GPUs. Adjust them based on your specific model size, hardware generation, and SLA requirements.

Warning-Level Alerts

  • TTFT p99 above 5 seconds sustained for 5 minutes. This indicates emerging latency problems, typically declining cache hit rates or increasing queue depth. Investigate promptly but do not necessarily page anyone at 3am.
  • KV cache hit ratio below 40% sustained for 10 minutes. Your routing layer may not be directing requests to pods with cached prefixes effectively. Check EPP scoring weights and pod traffic distribution.
  • GPU memory utilization above 90% sustained for 5 minutes. You are approaching the threshold where KV cache evictions begin cascading. Consider scaling horizontally or tuning your cache eviction policy before the situation degrades further.
  • Error rate above 1% sustained for 5 minutes. Occasional errors are expected in any distributed system, but sustained errors above one percent indicate a systemic issue that requires investigation.

Critical-Level Alerts

  • TTFT p99 above 10 seconds sustained for 3 minutes. Users are having a terrible experience and this is an active incident requiring immediate response.
  • Queue depth above 50 sustained for 5 minutes. The cluster cannot keep up with incoming request volume. Scale immediately or activate load shedding.
  • Pod restarts above 3 within a 15-minute window. Something is causing pods to crash-loop, most likely OOM kills from GPU memory exhaustion or a misconfigured model configuration.
# PrometheusRule for llm-d alerting
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: llm-d-inference-alerts
  namespace: llm-d
spec:
  groups:
    - name: llm-d.latency
      rules:
        - alert: HighTTFTWarning
          expr: |
            histogram_quantile(0.99,
              rate(vllm_request_ttft_seconds_bucket[5m])
            ) > 5
          for: 5m
          labels:
            severity: warning
          annotations:
            summary: "TTFT p99 exceeds 5s"
            description: "P99 time-to-first-token is {{ $value | humanizeDuration }}"
        - alert: HighTTFTCritical
          expr: |
            histogram_quantile(0.99,
              rate(vllm_request_ttft_seconds_bucket[5m])
            ) > 10
          for: 3m
          labels:
            severity: critical
          annotations:
            summary: "TTFT p99 exceeds 10s - active incident"
            description: "P99 TTFT at {{ $value | humanizeDuration }}. Users severely impacted."
    - name: llm-d.cache
      rules:
        - alert: LowCacheHitRate
          expr: |
            (sum(rate(vllm_cache_hits_total[5m])) /
            (sum(rate(vllm_cache_hits_total[5m])) +
             sum(rate(vllm_cache_misses_total[5m])))) * 100 < 40
          for: 10m
          labels:
            severity: warning
          annotations:
            summary: "KV cache hit rate below 40%"
            description: "Cache hit ratio is {{ $value }}%. Check EPP routing."
    - name: llm-d.resources
      rules:
        - alert: GPUMemoryPressure
          expr: DCGM_FI_DEV_FB_USED / DCGM_FI_DEV_FB_TOTAL * 100 > 90
          for: 5m
          labels:
            severity: warning
          annotations:
            summary: "GPU memory utilization above 90%"
        - alert: HighQueueDepth
          expr: sum(vllm_num_requests_waiting) > 50
          for: 5m
          labels:
            severity: critical
          annotations:
            summary: "Inference queue depth exceeds 50"
        - alert: PodCrashLoop
          expr: |
            increase(kube_pod_container_status_restarts_total{
              namespace="llm-d"
            }[15m]) > 3
          for: 1m
          labels:
            severity: critical
          annotations:
            summary: "Inference pod restarting repeatedly"
        - alert: HighErrorRate
          expr: |
            sum(rate(vllm_request_errors_total[5m])) /
            sum(rate(vllm_request_total[5m])) * 100 > 1
          for: 5m
          labels:
            severity: warning
          annotations:
            summary: "Request error rate above 1%"

To avoid alert fatigue, use inhibition rules to suppress warning-level alerts when a critical alert is already firing for the same component. If HighTTFTCritical is active, there is no value in also firing HighTTFTWarning or LowCacheHitRate. Group related alerts so that a single notification contains all the context an operator needs to diagnose the issue. And always include a runbook link in the alert annotations that explains not just what the alert means but what specific steps to take in response. An alert without a runbook is just noise that erodes operator confidence in the monitoring system.

Distributed Tracing for Inference

Metrics tell you that something is wrong. Traces tell you where it went wrong and why. For llm-d, distributed tracing is essential because a single inference request traverses multiple components, each with its own latency contribution and failure modes. Understanding the full request lifecycle requires end-to-end traces that correlate events across the Gateway, EPP, and vLLM inference pods into a single coherent narrative.

The request journey through an llm-d cluster has seven distinct stages, and each stage should be represented as a span in your distributed trace. The first span captures the Gateway, where TLS termination, authentication, rate limiting, and request validation occur. This typically contributes 1-5ms of latency, but misconfigured rate limiters or certificate validation issues can inflate it dramatically. The second span covers EPP scoring, where the Endpoint Picker Policy evaluates every candidate pod based on cache state, load, queue depth, and health status. This is usually sub-millisecond but becomes significant when the candidate pool is large or the scoring function involves complex cache lookup operations. The third span captures pod selection and request forwarding, including any network latency between the EPP and the selected inference pod.

The fourth span is where most of the latency lives: prefill computation. This span should be annotated with the number of input tokens, the number of tokens served from the KV cache, and the number of tokens that required fresh computation from scratch. This annotation is critical because it lets you distinguish between "this request was slow because it had a 50K-token context" and "this request was slow because it missed the cache and had to recompute 50K tokens that should have been cached." The fifth span covers KV cache transfer, which is relevant when llm-d performs disaggregated prefill/decode and transfers KV cache state between specialized prefill pods and decode pods over NVLink or RDMA. The sixth span covers the decode phase, where tokens are generated autoregressively one at a time. The seventh span captures response streaming back through the Gateway to the client.

OpenTelemetry is the industry standard for instrumentation. vLLM supports OpenTelemetry natively, and the llm-d control plane components can be instrumented using the OpenTelemetry SDKs for Go and Python. The key configuration requirement is ensuring that trace context is propagated across component boundaries using W3C Trace Context headers, which allows a single trace to span the entire request lifecycle from Gateway ingress to client response.

# OpenTelemetry Collector config for llm-d tracing
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

processors:
  batch:
    timeout: 5s
    send_batch_size: 1024
  attributes:
    actions:
      - key: deployment.environment
        value: production
        action: upsert

exporters:
  jaeger:
    endpoint: jaeger-collector.observability:14250
    tls:
      insecure: true

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch, attributes]
      exporters: [jaeger]

The real power of tracing emerges when you correlate traces with metrics. A TTFT spike on your Grafana dashboard becomes immediately actionable when you can click through to the specific traces that contributed to the p99 and see that they all missed the cache on a specific pod, or that they all spent 3 seconds waiting in the EPP queue because a pod health check was returning stale data. This correlation between aggregate metrics and individual request traces is what transforms monitoring from "something is wrong somewhere" to "here is exactly what went wrong, on which component, and why."

GPU Topology Monitoring

Modern GPU servers are not homogeneous compute pools. The physical interconnect topology between GPUs dramatically affects inference performance, and llm-d's scheduler is topology-aware for good reason. Understanding and monitoring GPU topology is essential for diagnosing performance anomalies that have no obvious cause at the application layer and that standard Kubernetes metrics cannot even represent.

The critical distinction is between NVLink and PCIe interconnects. NVLink provides over 600 GB/s of bidirectional bandwidth between directly connected GPUs on the same NVLink domain, while PCIe Gen5 x16 provides approximately 32 GB/s. That is nearly a 20x difference in communication bandwidth. When tensor parallelism is used to split a large model across multiple GPUs, the all-reduce operations that synchronize intermediate activations between GPUs are extremely bandwidth-intensive. If those GPUs are connected via NVLink, the communication overhead is negligible relative to the computation time. If they are connected via PCIe because the scheduler placed tensor-parallel shards on GPUs in different NVLink domains, the communication overhead can dominate the computation time entirely, especially for smaller batch sizes where the compute-to-communication ratio is already unfavorable.

NUMA (Non-Uniform Memory Access) topology adds another dimension of complexity. On multi-socket servers, GPUs are physically attached to specific CPU sockets via PCIe root complexes, and accessing GPU memory or performing DMA transfers from the wrong NUMA node incurs additional latency due to inter-socket communication over UPI or Infinity Fabric. llm-d's scheduler should place tensor-parallel groups on GPUs that share the same NVLink domain and the same NUMA node. Monitoring confirms that this placement is correct and alerts when topology violations occur, for example when a pod restart causes Kubernetes to reschedule a pod onto a node where the available GPU is not NVLink-connected to its tensor-parallel peers.

The DCGM (Data Center GPU Manager) exporter is the standard tool for exposing GPU-level metrics to Prometheus. Beyond the basic utilization and memory metrics, DCGM exposes NVLink-specific counters that are invaluable for topology monitoring and hardware health assessment.

# Key DCGM metrics for topology monitoring
DCGM_FI_DEV_NVLINK_BANDWIDTH_TOTAL       # NVLink aggregate throughput
DCGM_FI_DEV_NVLINK_CRC_FLIT_ERROR_COUNT  # NVLink CRC flit errors
DCGM_FI_DEV_NVLINK_CRC_DATA_ERROR_COUNT  # NVLink CRC data errors
DCGM_FI_DEV_NVLINK_REPLAY_ERROR_COUNT    # NVLink replay errors
DCGM_FI_DEV_PCIE_TX_THROUGHPUT           # PCIe transmit throughput
DCGM_FI_DEV_PCIE_RX_THROUGHPUT           # PCIe receive throughput

NVLink errors deserve special and immediate attention. A rising count of CRC errors or replay errors on an NVLink connection can indicate a failing GPU, a loose physical connection in the NVLink bridge, or thermal issues causing signal integrity degradation. These errors cause retransmissions that reduce effective bandwidth and increase latency jitter in ways that are nearly impossible to diagnose from application-level metrics alone. Left unmonitored, a degrading NVLink connection can cause intermittent latency spikes that appear random at the application layer but have a clear hardware root cause. Alert on any non-zero rate of NVLink errors and investigate immediately. Hardware issues do not fix themselves, and they invariably get worse over time until the link fails entirely.

llm-d uses topology information actively for scheduling decisions. When placing a new model replica, the scheduler prefers GPU groups that are fully connected via NVLink within the same domain. When performing KV cache transfers for disaggregated prefill/decode architectures, the scheduler prefers transfer paths that traverse NVLink rather than PCIe to minimize transfer latency. Monitoring these scheduling decisions and correlating them with observed inference latency validates that topology-aware scheduling is actually delivering the expected performance benefits rather than just adding complexity.

Cost Observability

GPUs are the most expensive resource in your infrastructure stack. An H100 SXM instance costs roughly $30-40 per hour on major cloud providers, and when you are running a cluster of eight or sixteen of these machines, every single percentage point of utilization efficiency translates directly to thousands of dollars per month in savings or waste. Cost observability is not a nice-to-have feature for mature organizations. It is the metric layer that determines whether your LLM serving infrastructure is financially sustainable at scale or whether you are hemorrhaging money on underutilized hardware.

The fundamental cost metric for LLM inference is cost-per-token. The calculation is straightforward in principle: take your total GPU cost per hour, divide by the number of tokens generated per hour, and you get the marginal cost of producing each token. This number should be tracked continuously and broken down by model, by LoRA adapter, and by request type to identify where your GPU dollars are going and where they are being wasted.

# Cost-per-million-tokens calculation
# Assume $35/hr per H100 GPU, 8 GPUs in cluster
# GPU cost per second
gpu_cost_per_second = 8 * 35 / 3600  # $0.0778/second

# Tokens generated per second (from Prometheus)
# sum(rate(vllm_tokens_generated_total[5m]))
# Example: 2,400 tokens/second

# Cost per million tokens
# ($0.0778 / 2400) * 1,000,000 = $32.41 per million tokens

The transformative impact of cache-aware routing, and the core economic argument for llm-d's existence, becomes quantifiable through cost observability. When a request hits the KV cache, it skips the prefill phase entirely. Prefill is the most compute-intensive part of inference, often consuming 10-50x more GPU cycles per token than the decode phase. By routing requests to pods that have already cached their prefix, llm-d eliminates redundant prefill computation, which directly reduces the number of GPU-seconds consumed per request, which directly reduces cost-per-token.

A practical example makes the magnitude clear: consider a RAG application that prepends a 10,000-token document context to every user query. Without cache-aware routing, every request pays the full prefill cost for those 10,000 document tokens regardless of which pod handles the request. With llm-d routing requests to pods that have already cached the document context, only the user query portion, perhaps 50-200 tokens, needs fresh prefill computation. That is a 98% reduction in prefill compute for the repeated prefix portion of each request. At scale, across thousands of requests per hour, this translates to running the same workload on 30-40% fewer GPUs, or equivalently, serving 50-70% more traffic on the same hardware footprint without any degradation in latency or quality.

Your cost dashboard should track three core dimensions. First, the raw cost per million tokens over time, decomposed by model and adapter. This is your unit economics and the number your finance team cares about. Second, the estimated cost savings from cache hits, calculated by comparing actual prefill tokens computed against the total tokens that would have required prefill without caching. This quantifies the dollar value of your routing intelligence. Third, GPU utilization efficiency, which is the ratio of time GPUs spend doing useful inference work versus time spent idle, in overhead, or performing redundant computation that caching should have eliminated. A well-optimized llm-d cluster should achieve 70-80% utilization efficiency. Below 50% means you are paying for GPU time that is not producing tokens, and you should investigate whether the bottleneck is scheduling inefficiency, memory management overhead, or simply insufficient request volume to keep the hardware busy.

Cost Impact: In production llm-d deployments with high prompt reuse patterns, cache-aware routing typically reduces cost-per-token by 30-50% compared to naive round-robin load balancing. For a cluster spending $100K/month on GPU compute, that represents $30K-$50K in monthly savings from routing intelligence alone.

Production Dashboard Design Patterns

Building dashboards is easy. Building dashboards that people actually look at during a 2am incident and can parse in seconds under stress is a fundamentally different challenge. The difference lies in applying established design patterns that account for how humans process visual information when they are tired, stressed, and under time pressure. When your cluster is actively degrading and users are complaining, the on-call engineer needs to understand the situation in seconds, not minutes spent scrolling through fifty undifferentiated panels.

The Four Golden Signals for LLM Inference

Google's Four Golden Signals framework, consisting of latency, traffic, errors, and saturation, adapts naturally to LLM inference, but the specific metrics that represent each signal are fundamentally different from traditional web services and must be carefully selected.

  • Latency is TTFT plus ITL. TTFT captures the initial wait that determines whether users perceive the system as responsive. ITL captures the streaming experience that determines whether the generation feels smooth or stuttery. Display both prominently on the overview dashboard, with TTFT as the primary signal and ITL as supporting context that aids in diagnosis.
  • Traffic is requests per second plus tokens per second. Requests per second tells you how many users are being served. Tokens per second tells you how much actual computational work the cluster is performing. These can diverge significantly in practice: a few long-context requests generate enormous token volumes while the request count stays low, and conversely, many short requests can produce high request rates with modest token throughput.
  • Errors are HTTP 5xx responses plus OOM kills. HTTP 5xx errors indicate application-level failures such as timeouts, model loading failures, or input validation errors. OOM kills indicate infrastructure-level failures where the GPU or system memory is exhausted. Track them separately because they have completely different root causes and different remediation strategies.
  • Saturation is GPU memory utilization plus queue depth. GPU memory saturation triggers KV cache evictions that cascade into latency degradation. Queue saturation indicates insufficient compute capacity for the current traffic load. Both are leading indicators of user-facing latency problems: they spike before TTFT degrades, giving you a window to intervene proactively before users notice.

Dashboard Hierarchy and the RED Method

Organize your dashboards in a strict drill-down hierarchy that mirrors the investigation workflow. The top-level dashboard shows the Four Golden Signals for the entire cluster in four to six large panels that are readable from across the room. If something looks wrong, clicking on the anomalous signal navigates to a second-level dashboard that shows the same signal decomposed by pod, by model, or by LoRA adapter. If a specific pod or model is misbehaving, clicking further navigates to a third-level dashboard showing that entity's detailed metrics including GPU-level counters from DCGM, per-request traces, and historical behavior patterns.

The RED method (Rate, Errors, Duration) works particularly well for per-model views in multi-model deployments. For each model or adapter served by your llm-d cluster, display the request rate, the error rate, and the TTFT distribution side by side. This immediately highlights which models are experiencing issues and which are healthy, and it doubles as a capacity planning tool: if one model's request rate is growing 20% month-over-month while others are flat, you know exactly where to invest in additional GPU capacity before the growth causes saturation.

Panel Organization, Time Ranges, and Visual Hierarchy

Use the F-pattern for panel layout: place the most critical information in the top-left corner, because eye-tracking research consistently shows that is where humans look first when scanning a page. For llm-d, the top-left panel should be TTFT p99. The panel to its right should be error rate. The second row should contain traffic and saturation panels. Detailed breakdowns and supporting metrics belong below the fold where they are available for investigation but do not compete for initial attention during triage.

For time ranges, default to 6 hours for the overview dashboard. This provides enough history to see trends and identify when a degradation started without overwhelming the display with noise. For active incident investigation, switch to 1-hour or 30-minute views to see fine-grained changes in real time. For weekly capacity planning reviews, use 7-day or 30-day views to identify growth trends and recurring traffic patterns. Include deployment annotations as vertical lines on all time-series charts so that metric changes can be immediately correlated with code, configuration, or model version deployments. Every time your team deploys a new model version, updates the EPP scoring weights, changes the KV cache eviction policy, or scales the cluster, that event should appear as a visible marker on the time axis of every dashboard.

Color coding must be consistent across every dashboard in your hierarchy. Green for healthy, amber for warning, red for critical. Use the exact same thresholds that your alerting rules use so that the dashboard colors match the alert states precisely. When a panel turns red on the overview dashboard, the operator should know with certainty that a corresponding alert has either already fired or is about to fire within the alert's for duration. This consistency between dashboard visuals and alert state builds trust in the monitoring system, which is the single most important factor in determining whether operators actually rely on dashboards during incidents or abandon them in favor of ad-hoc log searching and gut instinct.

Design Principle: Every panel on your dashboard should answer exactly one question. If you cannot state the question a panel answers in a single sentence, the panel is trying to do too much. Split it into focused sub-panels or move secondary information to a drill-down dashboard one level deeper in the hierarchy.

Bringing It All Together

LLM inference observability is not a solved problem that you can address by installing a Helm chart and walking away. It requires understanding the unique computational characteristics of autoregressive token generation, the cascading failure modes of KV cache management under memory pressure, the cost dynamics of GPU compute at scale, and the topology constraints of modern multi-GPU servers. Standard Kubernetes monitoring will not save you. It will actively mislead you by reporting healthy dashboards while your users experience multi-second latencies caused by problems that your monitoring stack cannot even represent.

The observability stack described in this guide, combining Prometheus for metrics collection, Grafana for dashboard visualization, OpenTelemetry for distributed tracing, DCGM for GPU topology and hardware health, and PromQL alerting rules calibrated specifically for LLM workloads, gives you genuine visibility into what your llm-d cluster is doing and why it is doing it. It exposes the metrics that actually determine user experience, like TTFT percentiles, KV cache hit ratio, and cost-per-token, rather than the metrics that are easy to collect but misleading to interpret, like aggregate CPU utilization and pod count.

Start with the metrics foundation. Instrument your vLLM pods with the ServiceMonitor configuration provided above. Build the four essential dashboards: cluster overview, pod detail, cache performance, and cost analysis. Set up the alerting rules with thresholds calibrated for your specific hardware and SLA requirements. Then add distributed tracing to understand the per-request lifecycle and diagnose routing decisions. Then add GPU topology monitoring to catch hardware degradation before it cascades into application-level failures. Each observability layer builds on the previous one, and each layer gives you visibility into a class of problems that the layers below cannot detect. The gap between collecting data and understanding your system is the difference between monitoring and observability. For LLM inference at scale, genuine understanding is not optional.
