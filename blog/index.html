What Happens When You Hit /v1/chat/completions

When a request arrives at your llm-d cluster, it doesn't get dumped onto a random GPU. Unlike traditional API proxies that treat all backends identically, llm-d runs an intelligent routing layer that makes real-time decisions based on cache state, pod health, and current load.

Here's the journey every request takes:

  1. Gateway API receives the HTTPS request at your cluster's edge
  2. EPP (Endpoint Picker Policy) scores every healthy pod based on four factors
  3. The winning pod handles the request, leveraging its KV cache if possible

Each layer has a specific job. The Gateway handles authentication and rate limiting. The EPP handles routing intelligence. The vLLM pods handle inference. No single component tries to do everything.

The Gateway: Your Cluster's Front Door

The Gateway API is your cluster's entry point. It's based on the Kubernetes Gateway API standard, which means you're not locked into proprietary infrastructure. TLS termination, rate limiting, and authentication all happen here before the request reaches the routing layer.

The Gateway uses an HTTPRoute with custom backend references pointing to the EPP. This is where llm-d diverges from traditional load balancers: instead of round-robin or least-connections, the Gateway delegates routing decisions to a policy-driven component that understands LLM workloads.

Key Point: The Gateway doesn't decide which pod gets the request. It enforces security and compliance, then hands the routing decision to the EPP.

The EPP: The Brain Behind Every Routing Decision

EPP stands for Endpoint Picker Policy. It's the component that makes llm-d intelligent. The EPP maintains a real-time model of every pod in your cluster: which prompts they've cached, how many requests they're handling, how deep their queues are, and whether they're healthy.

When a request arrives, the EPP scores every candidate pod using four factors:

  1. Prefix cache match length — How many tokens of the incoming request are already cached on this pod (highest weight)
  2. Pod load — num_requests_running / capacity
  3. Queue depth — num_requests_waiting
  4. Health status — Is the pod healthy, degraded, or failing

A Concrete Scoring Example

Let's say you have three pods and a request comes in with a 32K-token context. Here's how the EPP might score them:

  • Pod A: 28K tokens cached, 2/8 requests running, 1 queued, healthy → Score: 950
  • Pod B: 0 tokens cached, 1/8 requests running, 0 queued, healthy → Score: 400
  • Pod C: 15K tokens cached, 4/8 requests running, 3 queued, healthy → Score: 620

Pod A wins. Even though it's handling more requests than Pod B, the 28K-token cache match is worth far more than the slight load difference. The request goes to Pod A, which skips prefilling those 28K tokens and jumps straight to generating the response.

Why cache match is weighted highest: Avoiding redundant prefill is the single biggest latency win in LLM inference. A cache hit can reduce time-to-first-token by 40-50x.

KV Cache: The Hidden Accelerator

To understand why prefix caching matters so much, you need to know what a KV cache is.

During the attention mechanism in a transformer model, the model computes key-value pairs for every token in the context. These pairs represent "what this token pays attention to" and are expensive to compute — but once computed, they can be reused if the same prefix appears in a future request.

vLLM implements automatic prefix caching. When a request comes in, vLLM checks if any prefix of the prompt matches something it's already processed. If there's a match, it reuses the cached KV pairs and only computes new ones for the novel portion of the prompt.

Think of it as a lookup table for work already done. If you've already processed "Summarize this legal document: [30K tokens of contract]", and a new request comes in with the same contract plus "focusing on liability clauses", you only pay the cost of computing the new suffix.

Health-Aware Failover: When Things Go Wrong

Pods fail. GPUs run out of memory. Networks flake. llm-d is designed to route around failures gracefully.

The EPP continuously monitors pod health using Kubernetes liveness and readiness probes. Each pod reports its status:

  • Green: Healthy, accepting requests
  • Amber: Degraded (high load, slow responses), but still alive
  • Red: Unhealthy (probe failures, OOM events, GPU errors)

When a pod transitions to red, the EPP immediately removes it from the candidate pool. In-flight requests may be retried on healthy pods if the failure is detected early enough. The KV cache on the failed pod is lost, but the EPP knows this and won't route subsequent requests there until the pod recovers.

Graceful degradation happens at the amber stage. A pod under heavy load gets a lower score, so new requests flow to healthier pods. This prevents a single pod from becoming a hotspot and cascading into full failure.

Operational Benefit: You can watch health transitions in real-time using the interactive sandbox. This is observable infrastructure — no more black-box load balancers where you guess why requests are slow.

Seeing It All in Action

The interactive sandbox lets you see every routing decision llm-d makes. You can watch requests flow through the Gateway, see the EPP score each pod, and observe how cache state influences routing.

Each scenario demonstrates a different aspect of llm-d's routing intelligence:

  • Cached vs. Cold Start: Send the same prompt twice and watch the second request route to the pod that cached it
  • Load Balancing: Fire a burst of traffic and see how the EPP distributes load across healthy pods
  • Failover: Kill a pod mid-request and watch the EPP route new traffic around it
  • Cache Drift: Send divergent prompts and observe how cache affinity breaks down when contexts no longer overlap

The sandbox is fully client-side — no backend, no API keys, no telemetry. It's a pedagogical tool, not a sales demo. Try the burst traffic scenario to see load balancing in action, or the failover scenario to watch health-aware routing respond to pod failures.

What This Means for Platform Engineers

If you're running LLM inference at scale, llm-d changes the economics:

  • Fewer GPUs needed: Cache-aware routing means better utilization of the GPUs you already have. You're not paying for redundant prefill across replicas.
  • Observable routing: Every routing decision is exposed via metrics and logs. You can trace why a request went to a specific pod, debug latency spikes, and optimize cache hit rates.
  • Standard Kubernetes primitives: llm-d uses Gateway API, HTTPRoute, and standard Kubernetes health probes. Nothing proprietary. You can swap components, integrate with existing observability stacks, and stay on the CNCF stack.
  • Operationally simpler: Health-aware failover means you don't need external circuit breakers or retry logic. The EPP handles it.
Key Insight: llm-d makes LLM routing observable and intelligent. You're not guessing why requests are slow — you're watching the routing decision in real-time and tuning the weights based on your workload.
